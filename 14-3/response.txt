================

GCC Trunk Installation
-----------------------
Installing a GCC trunk build with -freflection support proved extremely difficult
on macOS Apple Silicon (aarch64-apple-darwin25). Two native approaches were tried:

1. Homebrew gcc --HEAD: Homebrew's formula attempted a full bootstrap build from
   the latest GCC trunk commit (HEAD-5dc2a8a). The build failed during the
   stage1-bubble phase. Homebrew itself notes that --HEAD formula failures are
   "expected behaviour" for unstable upstream snapshots.

2. Manual build from source: Cloned GCC trunk (gcc.gnu.org/git) and attempted an
   out-of-tree configure/make. The build system rejected "aarch64-apple-darwin25"
   as an unsupported target in gcc/config.gcc (macOS 26 / darwin25 is too new for
   the current trunk). Overriding --build=aarch64-apple-darwin24.5.0 bypassed
   that check but the stage1 sub-configure then failed with uint64_t detection
   errors caused by clang/macOS SDK incompatibilities with GCC's autoconf scripts.

Since -freflection support in mainline GCC trunk is in progress but not yet
fully stable for all platforms, the reflection-specific changes are described
below but cannot be compiled locally. The non-reflection C++ improvements have
been applied directly to the source code.


Changes Made to xmlDataBind
-----------------------------

The following features from earlier in the lecture were added to the codebase.
All changes compile with C++20 (e.g., g++-15 -std=c++20) without -freflection.

1. General Value Tag Type with Deduced Non-Type Template Parameter (C++17)
   File: variadic_examples.h

   Added:
       template<auto V>
       struct vt {
           static auto constexpr value = V;
       };

   The existing codebase already has event_t<parser::event_type e>, which is
   a tag type specific to one enum. vt<V> generalizes this to any compile-time
   value using the C++17 deduced auto non-type template parameter. The lecture
   shows that with C++26 reflection, vt enables automatic enum_variant creation:

       template<Enumeration E>
       using enum_variant =
         [: substitute(^^std::variant,
              enumerators_of(^^E) | views::transform(make_vt)) :];

   vt is now in place so that when -freflection becomes available, enum_variant
   can be added with no further changes to vt itself.

2. consteval Typelist Size Accessor (C++20)
   File: variadic_examples.h

   Added:
       template<typename T>
       consteval int typelist_size() { return Length<T>::value; }

   The existing code uses constexpr extensively for template metaprogramming
   values. consteval goes further: the function can ONLY be called in a constant
   expression context. This makes the compile-time intent explicit and prevents
   accidental runtime evaluation (unlike constexpr, which silently falls back
   to runtime when arguments are not constant expressions). This is the key
   distinction the lecture draws between constexpr and consteval.

3. Static Assert Using consteval (C++20)
   File: formatter.h

   Added:
       static_assert(typelist_size<all_types>() == 5,
           "all_types size changed; update visitor and formatter infrastructure");

   Uses the new consteval function to enforce at compile time that all_types
   stays in sync with the visitor/formatter infrastructure. If a new type is
   added to all_types, this fires immediately rather than producing subtle
   runtime failures.

4. Enumeration Concept (C++20)
   File: my_type_traits.h

   Added:
       template<typename T>
       concept Enumeration = std::is_enum_v<T>;

   The Enumeration concept directly prepares for the reflection-based
   enum_variant from the lecture. With -freflection:

       template<Enumeration E>
       using enum_variant =
         [: substitute(^^std::variant,
              enumerators_of(^^E) | views::transform(make_vt)) :];

   constrains the template to enumeration types rather than accepting arbitrary
   types silently. Without reflection the concept still documents intent clearly.

5. Visitable Concept (C++20)
   File: xmlbind.h (definition), formatter.h (use)

   Added to xmlbind.h (after type_visitor):
       template<typename T>
       concept Visitable = requires(T const& t, type_visitor const& v) {
           t.accept(v);
       };

   Changed in formatter.h:
       // Before:
       template<typename T>
       struct single_formatter_visitor : virtual public formatter_visitor_base { ... }

       // After:
       template<Visitable T>
       struct single_formatter_visitor : virtual public formatter_visitor_base { ... }

   The formatter infrastructure relies on every handled type implementing
   accept(type_visitor const&). Previously this was an unconstrained template.
   Constraining it with Visitable catches misuse at the point of instantiation
   with a clear error message rather than deep in the template expansion.


Reflection Features (Pending -freflection)
--------------------------------------------
The following improvements from the lecture require a working -freflection build
and are NOT yet applied but are architecturally prepared for by the changes above:

- enum_variant via reflection: replace the manual PETTHelper<index_sequence<...>>
  boilerplate in xmlbind.h with the clean reflection-based enum_variant using vt
  and enumerators_of(^^parser::event_type).

- deriveds_of / direct_bases_t via reflection: replace the compile-time
  typelist-based direct_bases_t in my_type_traits.h with the reflection-based
  version using meta::bases_of, eliminating the need to maintain all_types by
  hand.

- Factory population via reflection: replace the manual all_types typelist
  passed to parameterized_factory with deriveds_of<type_base>, so that adding
  a new XML schema type automatically wires it into every factory and visitor.
